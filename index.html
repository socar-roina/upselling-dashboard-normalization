집계 기준, 데이터 출처, 쿼리

정의

• 노출 고객 / 노출 횟수: 앱 이벤트 로그에서 page_name='upsell_bottomsheet', event_name='view', upselling_type=pfNudge 인 이벤트. 고객 수 = 고유 member_id, 노출 횟수 = (날짜 × 고객) 단위.
• 수락(PF0 전환): 노출된 고객이 같은 날 실제로 PF0(pf_type='0') 예약을 생성한 경우. 단순 바텀시트 클릭이 아니라 결제 결과 기준.
• 수락률 = 수락(전환) ÷ 노출 횟수.
• 노출 시점 보험: 이벤트 form_data의 pf_original (70=PF70, 30=PF30). PF0 넛지는 PF70/PF30 고객에게만 발화.

데이터 출처 (BigQuery)

• 이벤트(노출, 클릭): socar-data.app_web_log.socar_app_web_log (Firebase에서 BQ로 적재, 약 D+1 반영)
• 결제와 예약(PF0 전환, 보험매출): socar-data.socar_biz_profit.profit_socar_reservation (pf_type, _rev_pf, created_at_kst)
• form_data는 정상 JSON이 아니라 {pf_original:70,upselling_type:pfNudge} 형식(이스케이프 혼재)이라 JSON_VALUE 불가, REGEXP로 추출.

정규화 배포 시점 확인

• BQ 시간대별 집계상 5/21 13시 view 3건에서 14시 306건, 16시 977건으로 급증. ABT(66% 노출) 대비 일 노출이 약 1.5배(≈100/66)로 늘어 전체 100% 노출 전환 확인. 따라서 기준 시점은 2026-05-21 14:00 KST.

주의

• 6/1: 이벤트 로그 적재 누락(노출 169건만 기록, 예약량은 정상), 추세 해석에서 제외.
• 최근 1~2일: 결제 데이터 D+1 반영으로 수락(전환)이 일부 과소집계 가능.
• 바텀시트 클릭(pf0_option) 미사용 사유: 클릭 UU가 노출 UU의 약 97%로(기본 선택/렌더 시 발화) 실제 수락으로 보기 어려워, 결제 결과 기준으로 집계.
-- 일별 노출 / PF0 수락 (정규화 배포 후)
WITH v AS (
  SELECT DISTINCT DATE(event_at_kst) AS d, member_id,
    REGEXP_EXTRACT(form_data, r'pf_original[\\:]+([0-9]+)') AS pf_original
  FROM `socar-data.app_web_log.socar_app_web_log`
  WHERE page_name='upsell_bottomsheet' AND event_name='view'
    AND REGEXP_EXTRACT(form_data, r'upselling_type[\\:]+([A-Za-z]+)')='pfNudge'
    AND event_at_kst >= DATETIME '2026-05-21 14:00:00'
    AND member_id IS NOT NULL
),
r AS (
  SELECT DISTINCT DATE(created_at_kst) AS d, member_id
  FROM `socar-data.socar_biz_profit.profit_socar_reservation`
  WHERE pf_type='0' AND created_at_kst >= DATETIME '2026-05-21 14:00:00'
)
SELECT v.d,
  COUNT(DISTINCT v.member_id) AS exposed_uu,
  COUNT(DISTINCT IF(r.member_id IS NOT NULL, v.member_id, NULL)) AS accepted_pf0_uu
FROM v LEFT JOIN r ON v.d=r.d AND v.member_id=r.member_id
GROUP BY 1 ORDER BY 1;
